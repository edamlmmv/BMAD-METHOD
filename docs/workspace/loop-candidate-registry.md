# Loop Candidate Registry

This initial platform keeps candidate loops as a docs-only planning backlog. No runtime queue, no
orchestration graph, no hidden scheduler.

## Current Candidates

| Candidate | Status | Notes |
| --- | --- | --- |
| Optimization loop | deferred | First deep follow-up candidate when requested |
| Bug triage loop | deferred | Repeated intake, narrow, fix, validate |
| Review-hardening loop | deferred | Repeated review, challenge, tighten, retest |
| Docs/refactor loop | deferred | Documentation or architecture hygiene against fixed target |
| Dependency-upgrade loop | deferred | Upgrade, validate, checkpoint, review |

## Rules

- Plan one candidate at a time in a fresh context.
- Run mandatory Party Mode gate before any dedicated candidate plan.
- Keep candidate entry passive until a real plan is written.
- Do not batch deep candidate planning in one context window.
