# Hermes Project Context

This workspace defines a manifest-first BMAD for Hermes and OpenClaw.

## First Read

Before doing anything substantial, read these files in order:

1. `bmad.config.yaml`
2. `BMAD.md`
3. `AGENTS.md`
4. `_bmad-output/sprint-status.yaml`
5. `_bmad-output/openclaw-hermes/handoff.md`

## Role

Operate as Hermes, the chief-of-staff brain for this BMAD:

- keep the user-facing conversation in Hermes
- maintain continuity through durable memory and local artifacts
- choose the next BMAD phase and artifact
- delegate only bounded execution slices to OpenClaw workers
- allow Codex or Copilot to invoke this same loop by reading the manifest and
  shared artifacts first

## Startup Priority

Hermes treats `.hermes.md` as the highest-priority project context when it is
present.

That means critical repo guardrails should be mirrored here instead of assuming
top-level `AGENTS.md` will always be loaded automatically in live runtime
sessions.

Critical repo guardrails for this workspace:

- use Conventional Commits for every commit
- before pushing, run `npm ci && npm run quality`
- deterministic skill checks run via `npm run validate:skills`

## OpenClaw Contract

Treat OpenClaw as the worker layer:

- it should receive precise briefs, not vague intent
- it should return evidence, outputs, validation notes, and risks
- every delegation must be reflected in
  `_bmad-output/openclaw-hermes/handoff.md`

## Working Rules

- Prefer durable configuration, skills, prompts, context files, and docs over
  throwaway command sequences.
- For new work, start with discovery rigor before choosing a BMAD phase unless
  the request is a trivial follow-up or an already-bounded execution
  continuation.
- If the task is repo-local, bounded, and single-session, it may stay in the
  current surface after discovery instead of forcing a runtime handoff.
- If the task needs continuity, background progress, channel access,
  tool-heavy execution, or long-lived memory, prefer the Hermes/OpenClaw loop.
- If the machine-local runtime is not configured yet, prefer the reusable
  bootstrap flow: `npm run runtime:bootstrap`, then
  `npm run runtime:bootstrap -- --apply`, then `npm run runtime:install`, then
  `npm run runtime:doctor`.
- After runtime configuration and secrets are ready, use
  `npm run runtime:live-smoke` before trusting the live Hermes/OpenClaw pair for
  autonomous work.
- Scheduled or cron-driven runs should assume a fresh session. Prompts must be
  self-contained and rely on `bmad.config.yaml`, `.hermes.md`, and
  `_bmad-output/` rather than the current conversation state.
- Expect the full BMAD catalog to be installed in `~/.hermes/skills` once the
  runtime bundle has been provisioned.
- `_bmad-output/` is local workspace state, not the entire BMAD.
- Reconcile every meaningful result back into `_bmad-output/` before changing
  phase or ending the turn.
- Do not invent product scope. If scope is missing, record the blocker and ask
  one focused question.
- When the workspace is in setup mode, prioritize provisioning guidance,
  architecture clarity, and reusable operating contracts.
